# 智囊团与 Cross-Check 规则

<!-- detection_keywords: cross-check, 智囊团, brain-trust, 多角色, 验证, review, 评审 -->

## 🔴 L0 MUST（违反即失败）

### CC-001: 多角色 cross-check 是通用原则
所有开发任务都应采用多角色交叉验证（架构师/开发者/测试专家/Review 专家），不仅限于特定项目。
任何一个角色所做的事情都需要有其他角色进行对应的 cross-check，不能单一决策。

### CC-002: cross-check 未落地前产出必须用户确认
当前 cross-check 仅 prompt 层面模拟（单 LLM 自审），不算真正独立多角色验证。
在真正机制落地前，所有开发产出必须经用户确认后才算交付。

### CC-003: 智囊团必须多 subagent 多角色
单 subagent 多角色不行，需要多个独立 subagent 分别扮演不同角色（架构师/方案设计师/可行性评审），确保真正独立的视角。

## 🟡 L1 REQUIRED（项目要求）

### CC-004: 智囊团串行依赖+两轮评审模式（v1.2）
架构师先行（基于原始输入）→ 设计师+评审员Round1并行（设计师基于输入+架构产出，评审员Round1评审架构方案）→ 评审员Round2（评审设计方案）→ 汇总 subagent。
评审员基于原始输入对每个环节逐条指出问题。

### CC-005: 智囊团任务设计必须包含验收方法
每个任务除了执行方案，还必须有明确的验收检查项+验收命令+预期结果，可自动化执行。

### CC-006: 设计变更必须先通知用户确认
设计变更在实施之前必须先通知用户，等用户确认后再派 subagent 执行。不能发现偏差后直接改。

## 🟢 L2 RECOMMENDED（最佳实践）

### CC-007: 飞书通知带任务 ID 前缀
飞书消息带任务 ID 前缀（如 `[T-002]`），用户回复时带 ID 关联对应任务。

### CC-008: Review 产出统一上传飞书
所有需要用户 review 的产出，先上传飞书文档再通知用户，方便直接批注。
