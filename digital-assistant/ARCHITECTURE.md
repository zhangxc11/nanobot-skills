# Digital Assistant — 架构约束文档

> **本文档是调度器体系（dispatcher + 流程框架）自身的架构约束与设计哲学。**
> 
> 每条原则均来自实际踩坑教训或用户明确指示（V1~V6 共 42 条批注沉淀）。
> 
> **使用方式**：本文档是 dev-workflow 文档三件套中 ARCHITECTURE.md 的设计哲学章节。涉及本项目改造时，各角色按 dev-workflow 流程自然读到——不需要 dispatcher 代码做特殊注入。

---

## 〇、设计哲学（Design Philosophy）

> 以下是整个调度器体系的顶层设计理念，来自 V1~V6 共 6 轮方案迭代中用户的 42 条批注沉淀。
> 所有具体约束（第一~六章）都是这些哲学的具体体现。做设计决策时，先回到这里校验方向。

### P1. Prompt 引导优先，代码只做确定性框架

可靠性优先通过 prompt 引导 LLM 行为，Python 代码只做宏观框架（状态机流转、文件 I/O、格式校验）。
语义判断（任务分类、质量评估、方案评审）全部交给 LLM，不用关键词匹配或正则猜测。

### P2. Cross-check 是基本原则，不是兜底环节

任何 LLM 产出都必须有另一个 LLM 角色检查。这不是流程末尾的"审计关卡"，而是贯穿每个环节的方法论：
- Architect 出方案 → Code Review 检查实现一致性
- Developer 写代码 → Code Review 检查 + 单元测试自测
- Tester 测试 → Test Review 审查测试过程和报告
- Dispatcher 调度 → Auditor 审查 dispatcher session 记录
- 角色互检是具体实现方式，cross-check 是背后的通用设计理念

### P3. 角色基于 Skill 文档自主工作

每个角色的指导内容写在 `.md` 文件中（`scripts/roles/*.md`），Worker 启动后自己读取。
Dispatcher 不负责把角色 guidance 搬运到 prompt 里——Skill 文档不需要 prompt 展开，subagent 自己读取。
角色经验通过复盘机制沉淀回 Skill 文档，形成持续改进循环。

### P4. 流程模板化，类型由提交方指定

流程是命名化的 Pipeline Pattern（如 `standard-dev`、`cron-auto`），任务创建时绑定。
不由代码猜测任务该走什么流程——流程类型（flow_type）由任务提交方显式指定。

### P5. Dev 环境先行，元改造 = 正常开发

任何改造都先在 dev 环境验证。Dev 环境 = dev skill（symlink 隔离），不是独立进程。
调度器自身的改造也是正常的开发任务，遵循同样的流程，不需要特殊模式。

### P6. 演进优先，但不排斥重写

现有经实战验证的机制（状态机、Gate、打回）优先演进式改造。
但如果改造太复杂，重写也可以——演进 vs 重写不是硬约束，是务实选择。
核心判断标准：别过度打补丁，流程和产出质量 > iteration 计数。

### P7. Dispatcher 专注流程，适度判断

Dispatcher 主要负责流程编排（派发角色、检查产出、决定下一步），但可以做适度的结构性判断（如检查报告格式、判断是否需要打回）。
Dispatcher 不能自查自己（违反 cross-check），靠审计 Worker 检查执行完整性。

### P8. 通用机制优先，不为特定 case 写特定代码

Dispatcher 的目标是调度各种类型的任务（代码开发、文档整理、数据分析等），涉及不同仓库和项目。
不应为某个特定项目或仓库在 dispatcher 代码中添加特定逻辑（如 `if scope == "xxx" then 注入某文件`）。
而是通过通用约定（如 dev-workflow 文档三件套、角色 Skill 文档）让机制自然生效——任何项目只要遵循约定就自动适用。

## 一、路径与环境

### 1.1 Skill 自洽定位，禁止硬编码 skill 名称

- **原则**：Skill 内部通过 `__file__` 相对定位自身路径，不硬编码 `digital-assistant` 等 skill 名称。做到"放哪用哪"。
- **违反后果**：skill 被 symlink 到 dev 目录（如 `digital-assistant-dev`）后路径全部断裂，dev 环境无法工作。

### 1.2 使用 `absolute()` 而非 `resolve()`

- **原则**：路径计算一律用 `Path.absolute()`，不用 `Path.resolve()`。
- **违反后果**：`resolve()` 会穿透 symlink 还原为真实路径，导致 dev 环境的 symlink 隔离机制失效，dev 改动直接影响 prod。

### 1.3 共享数据通过 WORKSPACE 引用

- **原则**：`data/brain/` 等共享数据目录通过 `WORKSPACE` 环境变量或配置引用，不跟着 skill 目录走。
- **违反后果**：多个 skill 实例（prod/dev）各自维护一份数据，状态不一致，数据丢失。

## 二、内容组织

### 2.1 角色 Guidance 写在 .md 文件中，不硬编码在 Python 里

- **原则**：角色指导内容（guidance）写在 `scripts/roles/*.md` 文件中。Python 代码只负责调度逻辑（读文件、拼 prompt、调 API），`.md` 文件负责内容。
- **违反后果**：这是本文档诞生的直接原因。设计评审时明确说"guidance 在 .md 里"，实现时却把大段文本硬编码在 Python 函数的 f-string 中。导致：(1) 内容修改需要改代码+重启；(2) 非开发者无法维护 guidance；(3) 设计意图与实现严重脱节。

### 2.2 规则写在 `rules/*.md`，代码通过 rule_loader 加载

- **原则**：业务规则、检查清单等内容放在 `rules/*.md` 文件中，代码通过统一的 rule_loader 机制加载。
- **违反后果**：规则散落在代码各处，修改一条规则需要全局搜索，容易遗漏，无法让业务人员直接维护。

### 2.3 代码只做确定性的事，语义判断交给 LLM

- **原则**：Python 代码负责状态机流转、文件读写、格式校验等确定性逻辑。任何需要"理解含义"的判断（如判断任务类型、评估质量）都交给 LLM。
- **违反后果**：用关键词匹配、正则等方式做语义判断，覆盖率低、脆弱，边界 case 层出不穷。

## 三、Prompt 设计

### 3.1 Dispatcher→Worker prompt 长度确定可控

- **原则**：Dispatcher 发给 Worker 的 prompt 必须长度确定、可控，不因任务复杂度而膨胀。
- **违反后果**：复杂任务的 prompt 超长，token 浪费严重，甚至超出上下文窗口，Worker 表现不稳定。

### 3.2 Dispatcher 不做内容搬运工

- **原则**：设计文档、前序报告、历史记录等大块内容保留在文件中，prompt 只传文件路径引用，Worker 自己读文件获取内容。
- **违反后果**：Dispatcher prompt 膨胀（违反 3.1），且同一份内容在 prompt 链路中被反复复制，token 成本翻倍，还可能因截断导致信息丢失。

### 3.3 Worker prompt 中的路径动态生成

- **原则**：Worker prompt 中涉及的 shell 命令路径、文件路径等，从 `_path_config.py`（或等效配置）动态生成，不硬编码 skill 名称。
- **违反后果**：与 1.1 同源问题。prompt 中写死 `digital-assistant/scripts/xxx.py`，dev 环境下 Worker 执行的命令指向 prod 路径。

## 四、流程框架

### 4.1 Auditor 是框架层固定环节，不可豁免

- **原则**：Auditor（审计/复核）是框架层自动注入的环节，不在流程模板里定义，任何流程类型都不可跳过。
- **违反后果**：某些流程"忘记"加 Auditor，产出质量无人把关，错误直接交付给用户。

### 4.2 流程类型由任务提交方指定

- **原则**：`flow_type` 由任务提交方（用户或上游系统）显式指定，不由代码通过关键词匹配自动猜测。
- **违反后果**：代码用"包含'周报'→周报流程"这类规则猜测，误判率高，用户困惑为什么触发了错误的流程。

### 4.3 Cross-check 是方法论不是环节

- **原则**：Cross-check（交叉检查）是一种方法论——"任何 LLM 产出都应有另一个 LLM 调用来检查"，而不是流程中的一个固定步骤名。它可以体现在 Auditor 中，也可以体现在 Worker 内部的自检中。
- **违反后果**：把 cross-check 做成流程中的一个独立环节，导致流程僵化，且只在特定位置检查，其他位置的产出反而无人复核。

## 五、验收标准

### 5.1 E2E 测试在 Architect 阶段定义

- **原则**：端到端测试必须在 Architect 阶段就明确定义，包括：执行方式（CLI / API / UI）、环境（dev / prod）、数据流（真实 / mock）。
- **违反后果**：开发完成后才想测试方案，发现架构不支持某种测试方式，被迫返工或降低测试标准。

### 5.2 验收基于真实执行，不能纯靠 mock

- **原则**：最终验收必须基于真实执行（真实调用链路、真实数据流），不能仅靠 mock 或单元测试通过就算验收。
- **违反后果**：单元测试全绿，集成后立刻崩溃。Mock 掩盖了接口不匹配、数据格式不一致等问题。

### 5.3 Architect 同时产出设计方案 + 测试方案

- **原则**：Architect 的交付物必须同时包含设计方案和对应的测试方案，二者一起评审。
- **违反后果**：设计方案看起来完美但不可测试，或测试方案与设计脱节，验收时扯皮。

## 六、开发约束

### 6.1 开发隔离必须在 git 仓库内（feature branch）

- **原则**：所有开发工作在 git feature branch 上进行，禁止裸文件 cp 部署。
- **违反后果**：无法追溯改动历史，无法回滚，多人协作时互相覆盖，"上次能用的版本"找不回来。

### 6.2 Dev 环境通过 skill symlink 隔离

- **原则**：Dev 环境通过 symlink（如 `digital-assistant-dev` → dev 工作目录）隔离，不需要单独起进程。Prod 和 Dev 共享同一个 nanobot 实例，通过不同的 skill 路径区分。
- **违反后果**：为 dev 单独起进程，端口冲突、配置混乱、资源浪费，且 dev/prod 行为不一致。

### 6.3 部署前确认改动范围和分支状态

- **原则**：部署前必须确认：(1) 改动范围（前端/后端/配置）；(2) 代码已合并到目标分支；(3) 目标分支状态干净。
- **违反后果**：部署了未合并的代码、遗漏了前端改动、目标分支有未提交的脏文件，线上出现不可预期的行为。

## 附录：如何使用本文档

| 角色 | 使用时机 | 重点关注 |
|------|---------|---------|
| **Architect** | 出设计方案前 | 全文通读，确保方案不违反任何约束 |
| **Developer** | 写代码前 | 重点看"路径与环境"、"内容组织"、"Prompt 设计" |
| **Reviewer** | Code Review 时 | 逐条对照，作为 review checklist |
| **Auditor** | 验收时 | 重点看"验收标准"、"开发约束" |

> **本文档本身也应持续演进**：每次踩坑后，将教训提炼为新的约束条目补充进来，注明来源。
