# 人机协作模式

## 核心原则

**Agent 是高效的执行者，人是关键的决策者。**

长程问题解决中，agent 的优势是快速执行（读文件、改代码、跑测试），人的优势是全局判断（方向选择、优先级、风险评估）。最高效的模式是两者各司其职。

## Agent 自主推进的边界

### ✅ 可自主推进

| 类型 | 示例 |
|------|------|
| 数据收集 | 读日志、统计数据、grep 搜索 |
| 已确认方案的执行 | 按设计文档修改代码 |
| 测试和验证 | 运行测试、检查结果 |
| 文档更新 | 更新 STATE.md、写进展报告 |
| 复现工具构建 | 写 dump 逻辑、复现脚本 |
| 信息整理 | 汇总发现、整理问题清单 |

### 🛑 需要人工确认

| 类型 | 示例 | 原因 |
|------|------|------|
| 方向变更 | 从"逐个修 Fix"转向"综合重写" | 影响整体策略 |
| 架构决策 | "裁剪"vs"补入"的设计选择 | 有长期影响 |
| 生产操作 | merge 到 prod、重启服务 | 不可逆 |
| 优先级调整 | 暂停当前任务转做其他 | 影响资源分配 |
| 方案取舍 | 多个可行方案选哪个 | 涉及权衡判断 |

## 主动请求 Review 的时机

Agent 应在以下时刻主动暂停，向用户汇报并请求指导：

### 1. 定期检查点

每积累 3-5 个修复/发现后，主动做全局 review：

```
"目前已完成 Fix 1~5，发现以下模式：{模式}。
建议下一步：{方案 A} 或 {方案 B}。
是否继续当前方向，还是需要调整？"
```

### 2. 连续未进展

连续 2 轮迭代没有实质进展时：

```
"过去 2 轮尝试了 {方法}，但都未解决。
可能的原因：{分析}。
建议切换到 {新方法}，或者需要更多信息来判断。"
```

### 3. 意外发现

发现问题比预期复杂、或发现新的关联问题时：

```
"在修复 X 的过程中发现 Y 问题，严重度 {P0/P1/P2}。
Y 与 X 的关系：{关联分析}。
建议：{先修 Y 再修 X / 并行处理 / 记录后续处理}。"
```

### 4. 不可逆操作前

```
"准备将修改 merge 到 prod 分支。变更摘要：{摘要}。
验证状态：{已通过/部分通过}。
确认执行？"
```

## 避免局部最优

### 症状识别

Agent 陷入局部最优的典型表现：
- 每一步修改都"合理"，但整体方向偏移
- 累积的修改越来越复杂，但问题没有根本解决
- 反复在同一个区域修修补补
- 忽略了更简单的替代方案

### 应对策略

1. **定期全局 review**：不是"下一个 Fix 是什么"，而是"当前整体状况如何"
2. **记录否决方案**：在 STATE.md 中记录被否决的方案和原因，避免重复尝试
3. **人工"暂停键"**：用户说"先停一下，整体看看"时，agent 应完整汇报当前状态
4. **计数器意识**：如果 Fix 编号超过 5，强烈建议做一次综合 review

## 信息传递效率

### 向用户汇报的模板

```
## 当前状态
- 已完成：{列表}
- 进行中：{当前工作}
- 阻塞项：{需要人工决策的点}

## 关键发现
{最重要的 1-3 个发现}

## 建议下一步
{具体建议，如有多个方案则列出利弊}

## 需要确认
{具体问题}
```

### 从用户接收指令的理解

用户的简短指令背后往往有更深的意图：

| 用户说 | 可能的意图 |
|--------|-----------|
| "先不急改代码" | 暂停执行，做全局分析 |
| "用复现脚本试试" | 切换到离线调试模式 |
| "整体 review 一下" | 担心局部最优，需要全局视角 |
| "这个不需要你研究" | 任务的真正目的不是表面看到的 |
| "经验教训总结一下" | 需要可复用的知识沉淀 |
