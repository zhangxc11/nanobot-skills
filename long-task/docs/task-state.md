# 任务状态文件规范

## STATE.md 模板

```markdown
# {任务名称}

> **状态**: 进行中 | 已完成 | 已暂停
> **创建**: YYYY-MM-DD
> **最后更新**: YYYY-MM-DD HH:MM
> **关联 Session**: {session_id}（可能跨多个 session）

## 问题描述

{一段话概括问题}

## 已知问题清单

| # | 问题 | 严重度 | 状态 | 备注 |
|---|------|--------|------|------|
| 1 | ...  | P0     | ✅   | commit xxx |
| 2 | ...  | P1     | 🔧   | 进行中 |
| 3 | ...  | P2     | ⏳   | 待处理 |

## 关键决策

| 决策 | 原因 | 日期 | 决策人 |
|------|------|------|--------|
| 综合重写而非逐个叠 Fix | 9 个 Fix 累积导致代码难以维护 | 03-22 | 用户 |
| Pass 3 补入优先 | 裁剪导致信息永久丢失 | 03-22 | 用户 |

## 当前进展

- {最近完成的工作}

### 正在进行
- {当前正在做的事}

### 下一步（新 session 从这里开始）

{如何验证问题已修复}
```

## 使用规范

### 更新频率

- **每轮迭代结束后**必须更新"当前进展"和"下一步"
- **发现新问题时**更新"已知问题清单"
- **做出重要决策时**更新"关键决策"
- **关键文件变更时**更新"关键文件"

### 跨 Session 恢复流程

```
新 session 开始:
1. 用户提及任务名 或 agent 从 MEMORY.md 找到任务引用
2. read_file("data/analysis/{task}/STATE.md")
3. 从"下一步"继续推进
4. 如需更多上下文 → read progress.md 或 findings/
```

### MEMORY.md 中的引用

在 MEMORY.md 的 Active Work 区维护任务引用：

```markdown
- **{任务名}** — {状态emoji} {一句话描述}
  - **STATE**: `data/analysis/{task}/STATE.md`
  - **下一步**: {从 STATE.md 复制}
```

这样即使 session 完全重新开始，agent 也能从 MEMORY.md → STATE.md 恢复完整上下文。
