> 问题定义 → 状态文件建立 → 迭代推进 → 验证 → 经验沉淀

## 1. 适用场景

**适用**：
- 预计 3+ 轮调试/验证迭代的 bug 修复
- 跨多文件、多模块的分析或重构
- 需要构建复现工具的问题
- 可能跨越多个 session 的长期任务

**不适用**：
- 单次对话可完成的简单问题
- 纯代码开发（用 `dev-workflow`）
- 批量并行任务（用 `batch-orchestrator`）

| 相关 Skill | 区别 |
|------------|------|
| **dev-workflow** | 侧重开发流程（需求→设计→编码→测试）；本 skill 侧重问题解决过程的管理 |
| **batch-dev-planner** | 侧重多需求并行编排；本 skill 侧重单一复杂问题的深度推进 |

## 2. 核心流程

```
Phase 1        Phase 2          Phase 3        Phase 4       Phase 5
问题定义  →  状态文件建立  →  迭代推进  →  验证收敛  →  经验沉淀
```

### Phase 1: 问题定义

**先全面扫描问题空间，不要看到第一个症状就开始修。**

- 收集所有相关数据（日志、错误信息、用户报告）
- 区分**症状**和**根因**——多个症状可能指向同一根因
- 列出所有已知问题，标注严重度和关联关系
- 评估问题复杂度，决定是否需要启用本 skill

> 💡 §66 教训：127 次 consolidation 调用分析发现 10 个问题，如果只修第一个发现的 bug，其余 9 个会持续静默失败。

### Phase 2: 状态文件建立

**第一时间创建持久化状态文件**，这是跨 session / 跨 consolidation 的记忆锚点。

```bash
# 建议路径 — 按主题分类存放
data/analysis/{topic}/{task-name}/
├── STATE.md          # 任务状态（必须）
├── progress.md       # 详细进展日志
├── repro.py          # 复现脚本（如适用）
└── findings/         # 中间发现物

# 示例：
# data/analysis/consolidation/v70-rewrite/STATE.md
# data/analysis/cron/partition-redesign/STATE.md
```

> ⚠️ **如果要写入 `data/analysis/`，必须先读 `data/analysis/_INDEX_目录结构与分类规则.md`，遵循分类规则放入对应子目录。** 不要直接往 `data/analysis/` 根目录堆文件。

STATE.md 格式见 [docs/task-state.md](docs/task-state.md)。

### Phase 3: 迭代推进

每轮迭代遵循：

```
分析 → 假设 → 修改 → 验证 → 记录
```

**关键纪律**：
1. **每轮结束更新 STATE.md** — 记录本轮结果、下一步计划
2. **每 3-5 轮做一次全局 review** — 暂停细节推进，审视整体方向
3. **超过 2 轮线上调试未解决 → 构建离线复现工具**
4. **遇到方向性决策 → 记录到 STATE.md 的 Decisions 区，等待人工确认**

### Phase 4: 验证收敛

- 所有已知问题都有对应的修复
- 回归测试通过（新修复不破坏已有功能）
- 在目标环境中实际验证（不只是本地测试）

### Phase 5: 经验沉淀

- 撰写经验教训文档（retrospective）
- 更新 MEMORY.md 中的相关条目
- 如有通用模式，考虑提炼为 skill 或工具

## 3. 跨 Session 支持

长程任务经常跨越多个 session（用户关闭对话、session 超时、切换设备等）。

### 3.1 STATE.md 是唯一真相源

每次新 session 接手时：

```
1. 读 STATE.md → 了解当前状态、已完成工作、下一步计划
2. 读 progress.md（如需要）→ 了解详细历史
3. 从 STATE.md 中的"下一步"继续推进
```

**STATE.md 必须始终保持最新**——它是跨 session 的接力棒。

### 3.2 Session 交接清单

当预感到当前 session 可能结束（消息数接近 memory_window、用户暂时离开）时：

1. ✅ STATE.md 已更新到最新状态
2. ✅ 当前进展和未完成的工作已记录
3. ✅ 下一步计划明确且可执行
4. ✅ 关键发现已写入持久化文件（不只存在于对话中）
5. ✅ MEMORY.md 已更新任务引用

### 3.3 跨 Session 的信息传递

| 信息类型 | 持久化方式 | 恢复方式 |
|----------|-----------|----------|
| 任务状态 | STATE.md | 新 session 首先读取 |
| 详细进展 | progress.md | 按需读取 |
| 关键决策 | STATE.md Decisions 区 | 新 session 读取后遵循 |
| 代码变更 | git commit | git log 查看 |
| 中间数据 | findings/ 目录 | 按需读取 |
| 全局记忆 | MEMORY.md | 自动加载 |

## 4. 单 Turn 工具调用限制应对

Agent 每个 turn（一次 LLM 推理）能调用的工具次数有限（通常由 `max_iterations` 控制）。长程任务容易在单个 turn 中耗尽工具调用配额。

详见 [docs/tool-limits.md](docs/tool-limits.md)，核心技巧摘要：

### 4.1 Subagent 分担工具密集型工作

**主 session 负责协调和决策，subagent 负责执行。**

```
主 session:  分析问题 → 拆解子任务 → spawn subagent → 等结果 → 决策
subagent:   读文件 × N → 分析 → 写报告 → 返回结果
```

适合委派给 subagent 的工作：
- 多文件读取和分析
- 文档撰写
- 数据收集和统计
- 代码修改（在明确方案后）
- 测试执行和结果收集

**不适合**委派的工作：
- 需要人工确认的决策
- 依赖前一步结果的串行推理链
- 需要实时调整策略的交互式调试

### 4.2 批量操作合并

```bash
# ❌ 低效：逐个读文件
read_file("a.py")  # turn 1
read_file("b.py")  # turn 2
read_file("c.py")  # turn 3

# ✅ 高效：一次 exec 批量读
exec("cat a.py b.py c.py")           # turn 1
exec("grep -rn 'pattern' src/")      # turn 1（并行调用）
```

### 4.3 及时返回，刷新轮次

**启动 subagent 后应立即停下来返回**，等 subagent 完成后激活新 turn，刷新工具轮次限制。

```
# ❌ 低效：启动 subagent 后继续消耗轮次
spawn(task="分析文件")
read_file(...)        # 继续消耗当前 turn 的轮次
exec(...)             # 还在消耗...

# ✅ 高效：启动后立即返回
spawn(task="分析文件")
→ 返回用户 "已启动分析，等待结果"
→ subagent 完成后触发新 turn，轮次重置
```

### 4.4 脚本封装复杂操作

将多步操作封装为脚本，一次 exec 完成：

```bash
# 封装为脚本
write_file("tmp/analyze.sh", """
  grep -c 'error' logs/*.log
  wc -l src/**/*.py
  git log --oneline -20
""")
exec("bash tmp/analyze.sh")  # 1 次工具调用完成 3 个操作
```

## 5. 调试方法论

### 5.1 线上观察 vs 离线复现

| 维度 | 线上观察 | 离线复现 |
|------|---------|---------|
| 周期 | 30-60 分钟/轮 | 3-5 分钟/轮 |
| 信息完整度 | 受框架层过滤 | 原始完整数据 |
| 可重复性 | 低（依赖运行时状态） | 高（固定输入） |
| 初始投入 | 零 | 30-60 分钟构建工具 |
| 适用阶段 | 问题发现初期 | 深入调试阶段 |

**经验法则**：超过 2 轮线上调试未解决 → 立即投入构建离线复现脚本。

### 5.2 Dump + 复现脚本模式

```
1. 在目标代码中加 dump 逻辑（保存完整输入数据到文件）
2. 触发一次目标操作，获取 dump 数据
3. 编写独立脚本，用 dump 数据直接调用目标 API
4. 在脚本中快速迭代验证
5. 确认修复后，移除 dump 逻辑，将修复应用到正式代码
```

### 5.3 框架报错不可尽信

多层调用链（应用 → 框架 → 代理 → API）中，错误信息会被逐层封装和转译。

**实操经验**：
- litellm 等框架可能将超时包装为 429 Rate Limit
- 代理层可能将 API 错误包装为不同的 HTTP 状态码
- **诊断时绕过框架，直接用 httpx/curl 访问 API 看原始响应**
- 复现脚本的额外价值：绕过框架层，看到真实错误

## 6. 人机协作模式

### 6.1 Agent 自主推进的范围

| 可自主推进 | 需要人工确认 |
|-----------|-------------|
| 数据收集和分析 | 重大方向变更 |
| 已确认方案的代码实现 | 架构级设计决策 |
| 测试执行和结果报告 | 涉及生产环境的操作 |
| 文档撰写和更新 | 多个可行方案的取舍 |
| 复现脚本构建 | 问题优先级调整 |

### 6.2 主动请求 Review 的时机

- 积累 3-5 个修复后，主动暂停请求全局 review
- 发现问题比预期复杂时，及时上报
- 调试方向可能有误时（连续 2 轮未进展）
- 即将进行不可逆操作前

### 6.3 避免局部最优陷阱

Agent 在长程任务中容易陷入**局部最优**——在当前可见的上下文中做出合理但全局次优的决策。

**应对**：
- 定期"退一步看全局"：当前方向是否仍然正确？
- 记录被否决的方案和原因（避免重复尝试）
- 人工设置检查点（如"先不急改代码，整体分析一下"）

## 7. 快速参考：检查清单

开始长程任务时对照：

- [ ] **问题定义**：全面扫描问题空间，列出所有已知问题
- [ ] **STATE.md**：创建任务状态文件，记录初始状态和计划
- [ ] **复杂度评估**：预计迭代轮数？是否需要复现工具？
- [ ] **工具策略**：哪些工作适合 subagent？哪些需要脚本封装？

每轮迭代后对照：

- [ ] **STATE.md 已更新**：本轮结果 + 下一步计划
- [ ] **进展已记录**：关键发现写入持久化文件
- [ ] **方向检查**：是否需要全局 review？是否陷入局部最优？

准备交接/结束时对照：

- [ ] **STATE.md 最新**：新 session 可直接接手
- [ ] **代码已提交**：git commit 保存所有变更
- [ ] **MEMORY.md 已更新**：任务状态反映到全局记忆
