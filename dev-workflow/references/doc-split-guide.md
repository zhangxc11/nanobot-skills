# 文档拆分升级指南

从简版（单文件）升级到复杂版（主文件 + 子目录）的完整指南。

## 何时升级

满足以下**任一**条件时，应考虑拆分：

| 指标 | 阈值 | 说明 |
|------|------|------|
| 单文件行数 | > 800 行 | agent 读取困难，context 占用大 |
| 单文件大小 | > 30 KB | 接近 read_file 默认限制 |
| 章节数量 | > 15 个 | 导航困难，索引价值凸显 |
| 项目 Phase 数 | > 10 个 | DEVLOG 历史堆积 |

**不需要三个文件同时拆分**。哪个文件先到阈值就先拆哪个。

## 拆分原则

### REQUIREMENTS.md 拆分
- **分组依据**：按 §编号范围（每 10 个一组）
- **子文件命名**：`requirements/s01-s10.md`、`s11-s20.md`...
- **主文件保留**：项目概述 + 全量索引表 + Backlog 区域
- **子文件格式**：头部索引表 + 原文完整搬入

### ARCHITECTURE.md 拆分
- **分组依据**：按功能模块（不按编号顺序）
- **子文件命名**：`architecture/core.md`、`frontend.md`、`api.md`...
- **主文件保留**：架构总览 + 模块索引表 + 章节完整索引
- **子文件格式**：模块描述 + 头部索引表 + 原文完整搬入

### DEVLOG.md 拆分
- **分组依据**：按 Phase 编号范围（每 10 个一组）
- **子文件命名**：`devlog/phase-01-10.md`、`phase-11-20.md`...
- **主文件保留**：状态总览 + 全量索引表 + **最近 3 个 Phase 完整正文**
- **子文件格式**：头部索引表 + 原文完整搬入

## 升级步骤

### Step 1: 创建子目录和归档文件
```bash
mkdir -p docs/requirements docs/architecture docs/devlog
```

### Step 2: 按分组规则切分原文
- 每个归档文件头部加"本文件索引"表
- 原文完整搬入，不做任何删减

### Step 3: 重写主文件
- 删除已归档的正文
- 添加全量索引表（编号 + 标题 + 状态 + 归档文件链接）
- 添加 HTML 注释形式的"文档组织说明"

### Step 4: 验证
- `wc -l` 对比拆分前后总行数（归档 + 主文件 ≈ 原文 + 索引开销）
- 确认索引表与归档文件内容一致
- 确认所有链接可访问

## 日常维护

### 添加新需求
1. 在最新的归档文件末尾追加正文
2. 更新归档文件头部索引表
3. 更新主文件索引表
4. 从 Backlog 中删除对应条目

### 添加新 Phase
1. 在 DEVLOG 主文件底部追加新 Phase 正文
2. 如果主文件中已有 4+ 个 Phase 正文，将最旧的移入归档文件
3. 更新全量索引表和状态总览表

### 添加新架构设计
1. 确定属于哪个模块文件
2. 在模块文件末尾追加
3. 更新模块文件头部索引和主文件章节索引

## 主文件 HTML 注释模板

每个主文件标题下方应包含一个 HTML 注释形式的"文档组织说明"，格式参见各项目的实际文件。
注释内容包括：文档结构说明、如何查找、如何添加、维护规则。
