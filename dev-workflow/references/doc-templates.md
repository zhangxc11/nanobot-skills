# 文档模板参考

各文档的结构规范和模板。按需 read 本文件获取详细格式。

---

## REQUIREMENTS.md — 需求文档

记录所有功能需求、Bug Issue、设计约束。是"要做什么"的唯一来源。

### 结构模板

```markdown
# {项目名} — 需求文档

> 状态：**活跃** | 最后更新：YYYY-MM-DD

## 一、项目概述

简要描述项目目标、背景、技术栈。

## 二、功能需求

### Phase N: {功能名称}

#### FN.1 {子功能}
- 描述
- 输入/输出
- 约束条件

### Issue 记录

#### Issue #N: {问题描述}
- **状态**: 🐛 待修复 / ✅ 已修复 (commit xxx)
- **现象**: ...
- **原因**: ...
- **修复方案**: ...
```

### 原则

- 新功能 → 新增 Phase 章节（按序编号）
- Bug/改进 → 在对应 Phase 下追加 Issue，或新建独立 Issue 章节
- 每个需求有明确的验收标准
- 非目标（scope out）也要写明，避免 scope creep

## ARCHITECTURE.md — 架构设计文档

记录技术方案、模块结构、接口设计、测试设计。是"怎么做"的参考。

### 结构模板

```markdown
# {项目名} — 架构设计文档

> 版本：V1.0 | 最后更新：YYYY-MM-DD

## 一、架构概览

### 项目结构
（目录树 + 各文件职责说明）

### 整体设计
（架构图 / 流程图 / 模块关系）

## 二、核心模块

### 模块 A
- 职责
- 接口定义
- 关键实现细节

### 模块 B
...

## 三、数据流

（消息/数据在各模块间的流转）

## 四、测试设计

### 单元测试
| 测试文件 | 覆盖模块 | 测试项数 |
|----------|----------|----------|
| test_xxx.py | 模块 A | N 项 |

### 集成/端到端测试
（如有）

## 五、部署

（构建、启动、配置说明）
```

### 原则

- 随代码改动同步更新，保持与实现一致
- 图表优于长文描述
- 测试设计与架构设计放在同一文档，保持关联
- 关键设计决策记录理由（ADR 风格）

## DEVLOG.md — 开发日志

开发过程的唯一真相源。新 session 从这里恢复上下文。

### 结构模板

```markdown
# {项目名} — 开发日志

> 本文件是开发过程的唯一真相源。每次新 session 从这里恢复上下文。
> 找到 🔜 标记的任务，直接继续执行。

## 项目状态总览

| 阶段 | 状态 | 分支 |
|------|------|------|
| Phase 1: xxx | ✅ 已完成 | feat/xxx → main |
| Phase 2: yyy | 🔜 进行中 | feat/yyy |

---

## Phase N: {功能名称} (YYYY-MM-DD) 🔜

### 任务清单
- [x] T1: 已完成的任务
- [x] T2: 已完成的任务
- [ ] T3: 🔜 当前任务
- [ ] T4: 待做任务

### 进展记录

#### TN 完成记录
- 改了什么
- 测试结果
- commit hash

---

## Phase N-1: {功能名称} (YYYY-MM-DD) ✅

（已完成的 Phase，保留记录供回溯）

---

*本文件随开发进展持续更新。*
```

### 原则

- 🔜 标记当前进行中的任务（全文件只有一个活跃 Phase）
- 任务清单用 checkbox `- [ ]` / `- [x]`
- 每完成一个子任务立即勾选并记录
- 包含 commit hash、测试结果等关键信息
- 已完成的 Phase 保留但不再修改
