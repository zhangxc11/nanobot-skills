# Stage 0: 需求盘点

> 在进入 Stage 1 规划之前，先对 todo 池进行系统性盘点，确保纳入排期的需求都已充分对齐。

## 1. 核心流程

```
todo list → 按对齐状态分层 → 按 group 分组展示 → 用户确认可排期清单 → 进入 Stage 1
```

### 1.1 获取待开发清单

```bash
todo list --status todo --format json
```

读取所有 `status=todo` 的待办，作为盘点输入。

> ⚠️ **严格只看 `status=todo` 和 `status=doing`**。`status=done` 表示**开发已完成**，绝对不要将 done 项目纳入盘点。即使 done 项目仍带有 `已对齐`、`方案已确认` 等 tag，这些只是历史标记，不代表"等待开发"。

### 1.2 按对齐状态分层

根据 todo 的 tag 信息，将需求分为 4 个层级：

| 层级 | 条件 | 状态 | 下一步 |
|------|------|------|--------|
| **L1** | tag 含 `已对齐` + `方案已确认` | ✅ 可直接排期 | 进入 Stage 1 归集 |
| **L2** | tag 含 `已对齐` 但无 `方案已确认` | ⚠️ 需补方案 | 与用户讨论技术方案后补 tag |
| **L3** | tag 含 `待对齐` | ❌ 需与用户对齐 | 发起对齐讨论 |
| **L4** | 无对齐相关 tag | ❓ 需先分类 | 先判断是否为有效需求 |

### 1.3 按 group 分组展示

将分层结果按 todo 的 `group` 字段分组，便于用户按模块/仓库维度审视：

```
## nanobot-core
- [L1] #42 重构 config 加载逻辑（已对齐 + 方案已确认）
- [L2] #45 支持多 worker 模式（已对齐，需补方案）

## web-chat
- [L1] #51 会话列表分页（已对齐 + 方案已确认）
- [L3] #53 深色模式（待对齐）

## 未分组
- [L4] #60 优化启动速度（无标记）
```

### 1.4 输出可排期清单

将 L1 需求汇总为「可排期清单」，呈现给用户确认：

- 列出每条需求的 ID、标题、group、预估复杂度
- 标注 L2/L3/L4 需求的数量和处理建议
- **用户确认后**，可排期清单作为 Stage 1 的输入

## 2. 盘点期间的操作

### 2.1 L2 需求补方案

对已对齐但缺方案的需求：
1. 读取 `todo show <id>` 获取完整描述
2. 与用户讨论技术方案
3. 方案确定后：`todo update <id> --add-tag 方案已确认`
4. 升级为 L1，纳入可排期清单

### 2.2 L3 需求对齐

对待对齐需求：
1. 向用户展示需求概要，确认是否仍然需要
2. 对齐完成后：`todo update <id> --remove-tag 待对齐 --add-tag 已对齐`
3. 继续补方案流程（→ L2 → L1）

### 2.3 L4 需求分类

对无标记需求：
1. 判断是否为有效的开发需求（vs. 备忘/想法）
2. 有效 → 添加 `待对齐` tag，走 L3 流程
3. 无效 → 与用户确认是否关闭

## 3. 触发时机

- **主动触发**：用户说「盘点一下待办」「看看有什么可以开发的」
- **自动建议**：当 L1 需求积累 ≥ 5 条时，建议用户启动批量开发
- **批量开发前**：每次进入 Stage 1 前，建议先跑一遍 Stage 0 确保清单最新

## 4. 与 Stage 1 的衔接

Stage 0 的输出是一份**用户确认的可排期清单**（L1 需求列表）。

进入 Stage 1 时：
1. 以可排期清单为输入
2. 进行依赖分析和 Plan 归集
3. 不再重新筛选需求（除非用户要求追加）
4. **再次确认清单中无 `status=done` 项目** — done 表示"开发已完成"而非"待开发"，这个错误已发生过两次（batch-20260313, batch-20260320），必须在进入 Stage 1 前二次检查
