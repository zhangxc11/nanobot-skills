# 前端人眼验收策略

> 前端改动必须经过**人工视觉确认**，自动化测试无法替代。
> 前端验收在 **Stage 3 统一验收阶段**按 Plan（组）进行，不打断开发流程。

---

## 1. 为什么前端需要人眼确认

1. 前端 bug 大多是视觉/交互层面的（滚动行为、标记泄露、样式溢出、动画异常）
2. `npm run build` 通过 ≠ 功能正确（编译只检查类型，不检查运行时行为）
3. 自动化测试覆盖不了"看起来对不对"——这是人眼不可替代的价值

**核心原则：前端改动在 Stage 3 验收时，必须 build → 启 dev 环境 → 用户人眼确认 → 明确反馈后才标记通过。**

---

## 2. 验收环境

前端验收在 **dev 环境**进行，不影响线上服务：

| 维度 | Prod 环境 | Dev 环境（验收用） |
|------|----------|------------------|
| 代码 | 线上仓库 | `~/.nanobot/workspace/dev-workdir/{repo}` 的 feature branch |
| 端口 | webserver 8081 | webserver **9081** |
| 日志 | `~/.nanobot/logs/` | `~/.nanobot/logs-dev/` |
| 访问 | `localhost:8081` | `localhost:9081` |

---

## 3. 验收流程（Stage 3 内）

```
所有 Plan 开发完成，进入 Stage 3
    ↓
逐 Plan 验收（按依赖顺序）
    ↓
对含前端改动的 Plan：
    ↓
  1. 在 dev-workdir 中 build 前端
     cd ~/.nanobot/workspace/dev-workdir/web-chat/frontend && npm run build
    ↓
  2. 重启 dev webserver（端口 9081）
    ↓
  3. 通知用户：
     - 改了什么（功能点列表）
     - 验收要点（预期行为 + 操作步骤）
     - 访问地址：localhost:9081
    ↓
  4. ⏳ 等待用户人眼确认（不能自行判断"应该没问题"）
    ↓
  ┌─ 用户确认通过 → 标记 Plan passed → 下一个 Plan
  └─ 用户发现问题 → 在 feature branch 上修复 → 重新 build → 再次请用户确认（循环）
```

### 关键约束

- **必须等用户明确反馈**后才能继续，不能跳过人眼确认
- 每次修复后都要重新 build + 重启 dev webserver
- 修复 commit message 格式：`fix: {plan-name} review round {R} — {问题描述}`
- 修复在 feature branch 上进行，不污染主分支

---

## 4. 不同改动类型的验收方式

| 改动类型 | 验收方式 | 是否需要人眼确认 | 备注 |
|----------|---------|:----------------:|------|
| **纯前端**（组件/样式/store） | build + 重启 dev webserver + 人眼 | ✅ **必须** | 最常见场景 |
| **纯后端**（API/逻辑） | 自动化测试 + 代码审查 | ❌ 不需要 | 除非有 UI 联动 |
| **前端 + worker 联动** | 重启 dev worker + dev webserver + 人眼 | ✅ **必须** | 需要完整 dev 环境 |
| **Gateway 相关** | 停 prod GW → 启 dev GW + 人眼 | ✅ **必须** | 见 [nanobot/dev-env.md — Gateway 验收](nanobot/dev-env.md#gateway-验收特殊流程) |

---

## 5. 编排建议

- 前端需求与后端需求**可以在同一个 Plan 中**（按组而非按类型划分）
- 但 Plan 内部的 Step 拆分时，建议**纯前端改动与 worker 侧改动拆到不同 Step**，方便定位问题
- 前端 subagent 的 prompt 中应包含**验收操作指南**（功能点 + 预期行为 + 操作步骤）
- 多个前端 Plan 可并行开发，验收时逐个确认

---

## 6. 验收项分类

| 类型 | 验证方法 | 是否需要人眼 | 示例 |
|------|---------|:----------:|------|
| **UI 视觉** | 访问页面观察 | ✅ | 样式、布局、动画、响应式 |
| **UI 交互** | 点击/滚动/输入操作 | ✅ | 按钮响应、滚动行为、键盘快捷键 |
| **数据展示** | 对比数据源 | ✅ | 列表内容、数字精度、时间格式 |
| **数据验证** | 查数据库/文件 | ❌ | SQL 查询、grep 日志 |
| **代码确认** | grep 关键代码 | ❌ | 确认超时值、默认参数 |
| **行为验证** | 触发特定场景 | ❌ | spawn 多个观察排队 |
| **测试覆盖** | 单元测试 | ❌ | 边界场景 |
| **前端回归** | 刷新页面全局检查 | ✅ | 确认后续改动没破坏已验收功能 |
