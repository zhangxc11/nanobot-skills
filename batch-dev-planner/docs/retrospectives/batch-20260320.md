## 6. Bug 发现与修复记录

### 6.1 §64 Silent Cleanup 无限循环 Bug 🐛

**发现时间**: 2026-03-21 10:43 (验收 dev 环境部署验证时)

**现象**: warning 注入 → LLM 写 session summary → silent cleanup 删除 3 条消息 → `_warned_this_turn = False` 重置 → 下一轮循环再次注入 warning → 无限循环，每轮消耗 ~13K tokens

**根因**: `loop.py` ~L831 silent cleanup 完成后执行 `self._warned_this_turn = False`，本意是"清理完毕，恢复正常状态"，但实际效果是允许同一 turn 内再次触发 warning 注入

**修复**: commit `921f9b2` — 删除 `_warned_this_turn = False` 这一行，cleanup 后保持 `_warned_this_turn = True`，阻止同一 turn 内重复注入

**验证**: 修复后 dev 环境测试 session `webchat_dev_consolidation_test2_1774062048` 运行 14 iterations / 39 steps 正常完成，warning 只触发一次

**反思**: 如果验收只做静态审查+测试而不做 dev 环境部署验证，此 Bug 不会被发现。**运行时行为类改动必须做部署验证**。

### 12.2 发现的问题

| # | 问题 | 根因 | 修复 |
|---|------|------|------|
| 1 | 飞书 ASR 参数全错 | SA 开发时未实测飞书 API，参数基于文档推测 | file_recognize + file_id 16位 + engine_type 16k_auto |
| 2 | Local ASR 超时无响应 | Event.wait() 不驱动 NSRunLoop，ObjC 回调永不触发 | NSRunLoop 泵送循环 |
| 3 | Local ASR 偶发空结果 | SFSpeechRecognizer isFinal 回调返回空字符串 | best_partial fallback |

| 5 | 直接在 main 分支开发 | 独立仓库未遵循 feature branch 策略 | 已记录，不回退 |

### 12.3 关键教训

1. **SA 开发外部 API 集成时必须有实测环节** — Plan ASR 的 SA 基于文档推测飞书 API 参数，4 个参数中 3 个有误。改进：SA prompt 中对外部 API 调用应要求实际调用验证。

2. **macOS 原生框架的 ObjC 回调机制需特别注意** — Python 通过 PyObjC 调用 macOS API 时，回调依赖 NSRunLoop 驱动，标准 threading 机制无效。这是 Python↔ObjC 互操作的已知陷阱。

3. **独立 skill 仓库也必须走 feature branch** — 虽然 feishu-parser 是独立仓库，但 batch-dev 流程应统一要求所有仓库走 feature branch。本次直接在 main 开发，导致无法通过 merge --no-ff 保留开发历史。

4. **语音 session 验收效率较低** — 通过语音消息逐步指导验收，每轮需要识别语音+理解指令+执行+反馈，比文字交互慢。但优点是用户可以在移动中完成验收。

5. **Git commit 整理应在开发阶段就规划好** — 6 个零散 commit 最终整理为 2 个，如果一开始就按 feat+docs 分组提交，可以省去 rebase 整理的时间。
